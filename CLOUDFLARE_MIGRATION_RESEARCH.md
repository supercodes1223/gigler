# Cloudflare Platform Reference for Gigler Migration

**Audience:** Gigler architecture/planning  
**Goal:** Inform a decision to migrate the Gigler Next.js + AWS Amplify Gen 2 SaaS to Cloudflare Workers + Agents SDK + Durable Objects + R2 + D1.  
**As of:** April 2026. All facts are sourced from the official Cloudflare docs and blog posts cited at the end of each section. Beta status is called out clearly with **⚠ BETA** flags.

---

## TL;DR — Top capabilities Gigler benefits from most

1. **Cloudflare Sandboxes (GA April 13, 2026)** — Real coding gigs. Persistent containers per gig with PTY, snapshots, code interpreters, secure credential injection, and active-CPU pricing. This is the single largest enabler for "agents that actually code."
2. **Browser Run + Stagehand/Playwright MCP** — Real browsing gigs. Headless Chrome with CDP, Live View, MCP integration; persist sessions in Durable Objects.
3. **Agents SDK + Email Service** — Email-native agents. `onEmail`, `replyToEmail`, secure reply routing with HMAC-signed headers, all on a Durable-Object-backed agent class.
4. **Workflows + Agents `waitForApproval()` + sub-agent fibers** — Long-running, multi-step gigs that survive eviction, support human-in-the-loop, and can sleep up to 365 days.
5. **R2 with zero egress** — Materially cheaper than S3 for any gig that produces large artifacts (videos, datasets, build outputs).

The economic story is dominated by **(a) zero R2 egress** and **(b) active-CPU sandbox pricing** vs. AWS Lambda's wall-clock invocation model. The architectural story is dominated by **per-tenant Durable Objects with embedded SQLite** replacing shared DynamoDB tables.

## 1. Next.js on Workers via @opennextjs/cloudflare

**What it is.** The OpenNext adapter (`@opennextjs/cloudflare`) deploys a Next.js app to Cloudflare Workers using the Next.js **Node.js runtime** (not Edge runtime). Builds your app, transforms the output to `.open-next/worker.js`, and serves static assets via the Workers `assets` binding. ([opennext.js.org/cloudflare](https://opennext.js.org/cloudflare), [developers.cloudflare.com/workers/frameworks/framework-guides/nextjs](https://developers.cloudflare.com/workers/frameworks/framework-guides/nextjs/))

**How you use it.**

- New project: `npm create cloudflare@latest -- my-next-app --framework=next`.
- Existing app: install `@opennextjs/cloudflare` + `wrangler`, add `open-next.config.ts` with `defineCloudflareConfig()`, set `wrangler.jsonc` `main: ".open-next/worker.js"`, `compatibility_flags: ["nodejs_compat"]`, `compatibility_date: "2026-04-17"` or later.
- Local: `npm run dev` (Next.js dev server) for iteration, `npm run preview` for `workerd` parity, `npm run deploy` to ship.
- Wrangler can auto-detect Next.js and generate config — `npx wrangler deploy` works with no config file.

**Supported features.** App Router, Pages Router, RSC, SSG, SSR, ISR, Server Actions, streaming, `next/after`, Middleware, Image Optimization (via Cloudflare Images), PPR, `'use cache'`, Turbopack. **Node.js Middleware (Next 15.2+) is NOT yet supported.**

**Limits.**

- Worker size limit: 3 MiB free / **10 MiB paid** (compressed). Big Next apps can hit this; check `wrangler` output for `gzip` size.
- Next.js 14 support is being dropped (Next.js team itself dropped it). Stick to 15 or 16.

**Migration gotchas from Amplify.**

- Amplify's `amplify_outputs.json` build-time injection has no equivalent — Workers uses `wrangler.jsonc` bindings + `cloudflare-env.d.ts` (generated by `wrangler types`). You need to rewrite anything that calls `Amplify.configure(outputs)` to instead read from `env.*` bindings.
- Amplify Hosting SSR runs on Lambda@Edge with no IAM credentials by default; Workers gives you bindings (R2, D1, DO, KV) directly with no creds plumbing.
- `next/image` optimization needs Cloudflare Images (paid product) for parity.

**Sources:** [OpenNext Cloudflare](https://opennext.js.org/cloudflare), [Workers Next.js guide](https://developers.cloudflare.com/workers/frameworks/framework-guides/nextjs/), [Workers limits](https://developers.cloudflare.com/workers/platform/limits/).

## 2. Agents SDK (`agents` npm package)

**What it is.** A TypeScript class-based SDK where each Agent runs on a Durable Object with its own SQLite DB, WebSockets, and scheduling. State is automatic and survives deploys/hibernation. Methods marked `@callable()` become typed RPC over WebSocket. ([developers.cloudflare.com/agents](https://developers.cloudflare.com/agents/))

**Core capabilities relevant to Gigler:**

### `onEmail(email: AgentEmail)` — inbound email hook
Triggered when Email Service routes mail to your agent. The `AgentEmail` interface exposes `getRaw()`, `reply()`, `forward()`, `setReject()`. Pair with `routeAgentEmail()` in your Worker's `email()` handler and `createAddressBasedEmailResolver()` (or `createSecureReplyEmailResolver()`) to dispatch.

```ts
class GigAgent extends Agent {
  async onEmail(email: AgentEmail) {
    const raw = await email.getRaw();
    const parsed = await PostalMime.parse(raw);
    if (isAutoReplyEmail(parsed.headers)) return; // RFC 3834 mail-loop guard
    await this.replyToEmail(email, {
      fromName: "Gigler",
      body: "Got it — working on your gig now.",
      secret: this.env.EMAIL_SECRET, // signs X-Agent-Name/X-Agent-ID HMAC-SHA256
    });
  }
}
```

For deferred replies (after a sandbox finishes, after approval), store sender info in state and call `sendEmail()` with `inReplyTo` to thread correctly.

### `schedule()` and `scheduleEvery()`
Persisted in SQLite, backed by Durable Object alarms. Survives restarts.
- `this.schedule(60, "method", payload)` — run in 60s.
- `this.schedule(new Date("..."), "method", payload)` — run at time.
- `this.schedule("0 8 * * *", "method", payload)` — cron (idempotent by default).
- `this.scheduleEvery(45, "method", payload)` — fixed interval, sub-minute precision, built-in overlap prevention.

### `runFiber()` — durable execution (April 2026)
PR #1256 (April 2026) unified fiber architecture into the base `Agent` class. `runFiber(name, fn)` registers a task in SQLite and **keeps the agent alive during execution**, so long-running LLM streams or sandbox-runs survive Durable Object eviction. `ctx.stash()` checkpoints state; `onFiberRecovered()` fires on restart. For chat agents, `durableStreaming = true` automatically wraps each chat turn — partial responses persist and `continueLastTurn()` resumes interrupted messages. ([cloudflare/agents PR #1256](https://github.com/cloudflare/agents/pull/1256), [Durable execution docs](https://developers.cloudflare.com/agents/api-reference/durable-execution/))

### `waitForApproval()` — human-in-the-loop
Pattern 1 in the HITL guide: pause a Workflow until `approveWorkflow(id, { reason, metadata })` or `rejectWorkflow(id)`. Timeout up to weeks. Use `this.schedule()` to send escalation reminders. ([HITL concepts](https://developers.cloudflare.com/agents/concepts/human-in-the-loop/))

### Tool loops
`AIChatAgent` + `streamText` from the Vercel AI SDK gives you the standard tool loop. Tools support `needsApproval: boolean | async fn` for per-call approval. Client-side tools (e.g., browser geolocation) defined without `execute` and handled by the React `useAgentChat({ onToolCall })` hook.

### Sub-agents
**Status: emerging primitive, not yet first-class API.** Today you implement sub-agents by exposing a tool that calls another `AIChatAgent` instance via RPC. There is an [open feature request](https://github.com/cloudflare/agents/issues/368) for first-class streaming + cancellation. **Project Think** (Cloudflare's next-gen agent platform) introduces sub-agents as a core primitive with isolated SQLite, typed RPC, durable execution, and tree-structured message sessions — but is still in early access. ([Project Think blog](https://blog.cloudflare.com/project-think/))

### State: Durable Object SQLite
`this.setState(...)` syncs to connected clients in real-time over WebSocket. Backed by the agent's embedded SQLite DB. The `useAgent()` React hook gives you `agent.stub.method()` for typed RPC and `onStateUpdate` for live state.

**⚠ BETA / fast-moving.** The Agents SDK is GA but still evolving rapidly (v0.6.0 RPC transport in Feb 2026, fibers in April 2026). Pin versions and read changelogs before each upgrade.

**Sources:** [Agents overview](https://developers.cloudflare.com/agents/), [Email API](https://developers.cloudflare.com/agents/api-reference/email/), [Schedule tasks](https://developers.cloudflare.com/agents/api-reference/schedule-tasks/), [HITL](https://developers.cloudflare.com/agents/concepts/human-in-the-loop/), [Durable execution](https://developers.cloudflare.com/agents/api-reference/durable-execution/).

## 3. Durable Objects with SQLite (vs DynamoDB)

**What it is.** Each Durable Object instance gets a private, embedded SQLite database with strongly-consistent transactional storage and optional Point-In-Time Recovery (30 days). Access via `ctx.storage.sql.exec(...)`, sync KV via `ctx.storage.kv.get/put/delete/list`, async KV via `ctx.storage.get/put`, and `setAlarm()`/`getAlarm()` for scheduled wakeups. ([SQLite-backed DO Storage](https://developers.cloudflare.com/durable-objects/api/sql-storage/))

**Key features.**

- **Per-DO SQL** with full SQLite semantics (CREATE TABLE, indexes, JOINs, transactions).
- **Atomic writes**: any series of writes without an `await` between them is automatically a transaction. `transactionSync(callback)` for synchronous SQL transactions.
- **Automatic write coalescing** + in-memory cache; `noCache: true` and `allowUnconfirmed: true` for tuning.
- **Point-In-Time Recovery**: `getBookmarkForTime(...)`, `onNextSessionRestoreBookmark(...)` — restore the entire DO (SQL + KV) to any moment in the last 30 days.

**Comparison to DynamoDB:**

| Concern | DynamoDB | DO SQLite |
|---|---|---|
| Data model | Wide-column, single-table | Per-DO relational SQLite (full SQL) |
| Query model | PK/SK + GSIs/LSIs | Full SQL, indexes, joins, aggregates |
| Multi-tenant isolation | Logical (PK prefix) | Physical (one DO per tenant/gig/user) |
| Cross-tenant queries | Native (scan/query) | **None** — must fan out across DOs (use D1 for cross-tenant) |
| Per-record size | 400 KB max | SQLite row, MB-scale fine; DO storage 1+ GB per object |
| Strong consistency | Optional, 2x cost | Always (single-writer per DO) |
| Cost | $1.25/million writes (1KB), $0.25/million reads | $1.00/million row-writes, **$0.001/million row-reads** |
| GSIs | First-class, costly | Use SQLite indexes; or D1 for global views |
| Transactions | TransactWriteItems (25 items max) | Full SQL transactions, unlimited items |

**Pricing (April 2026).**

- **Compute**: 1M req + 400k GB-s included on Paid; then $0.15/million requests + $12.50/million GB-s (128 MB allocation per DO). WebSockets: 20:1 incoming-message-to-billed-request ratio. Use **WebSocket Hibernation API** to drop duration charges to ~zero between messages.
- **SQLite storage**: 25B row-reads + 50M row-writes + 5 GB-month included on Paid; then $0.001/million reads, $1.00/million writes, $0.20/GB-month. **Storage billing started January 7, 2026.**
- Free plan supports SQLite-backed DOs (5M reads/day, 100k writes/day, 5 GB).

**Limits to plan around.** 128 MB memory per DO at runtime. KV value sizes: ~2 MiB per value. SQLite row: standard SQLite limits. `setAlarm()` counts as 1 row write. Empty SQLite DB ≈ 12 KB stored.

**Migration gotchas from DynamoDB.**

- **GSIs don't exist.** If your access pattern needs "find all gigs across all users where status=running", you cannot do that with one DO — fan out across DOs (slow) or maintain a denormalized D1 table.
- **Pick the partition key carefully.** The DO ID determines isolation and parallelism. Per-user is common; per-gig may be better for isolation but costs more in idle DOs.
- **Hibernation matters for cost.** A DO with an open WebSocket and no Hibernation API will incur duration charges 24/7. The hibernation example in the pricing docs cuts cost from ~$416/mo to ~$10/mo for the same workload.
- **No cross-DO queries.** Reads in one DO cannot transact with writes in another. Use Workflows or a coordinator DO if you need this.

**Sources:** [SQLite-backed DO storage](https://developers.cloudflare.com/durable-objects/api/sql-storage/), [DO pricing](https://developers.cloudflare.com/durable-objects/platform/pricing/), [Storage billing changelog](https://developers.cloudflare.com/changelog/2025-12-12-durable-objects-sqlite-storage-billing/).

## 4. D1 (vs DO SQLite vs KV)

**What it is.** D1 is Cloudflare's managed serverless SQLite — designed for **horizontal scale-out across many small (10 GB) databases**, queried from any Worker. Includes Time Travel (point-in-time recovery up to 30 days), read replication, and a Worker binding. ([developers.cloudflare.com/d1](https://developers.cloudflare.com/d1/))

**Pricing.** Same per-row metering as DO SQLite: $0.001/million rows read, $1.00/million rows written, $0.75/GB-month storage. Free tier: 5M reads/day, 100k writes/day, 5 GB total. **No egress charges.**

**When to use which (decision tree):**

| Pattern | Best choice |
|---|---|
| Per-user / per-tenant / per-session state with frequent reads/writes from a single owner | **DO SQLite** |
| Real-time state synced to live WebSocket clients | **DO SQLite** (built-in `setState` sync) |
| Cross-tenant analytical queries, leaderboards, search across all users | **D1** |
| Rarely-read config, feature flags, small lookups, idempotency keys | **Workers KV** |
| Hot session cache | **Workers KV** (or DO SQLite for write-heavy) |
| Append-only audit log per tenant | **DO SQLite** |
| Vector search / embeddings | **Vectorize** (separate product) |

**Key D1 gotcha.** Reads are billed by **rows scanned**, not rows returned. A `SELECT * FROM users WHERE email = ?` without an index on `email` scans the whole table. Always add indexes for filtered columns.

**Migration gotcha.** Coming from DynamoDB single-table design, you'll often want to split into both: D1 for global metadata (users, gigs, billing) and DO SQLite for per-gig live state.

**Sources:** [D1 pricing](https://developers.cloudflare.com/d1/platform/pricing/), [D1 docs](https://developers.cloudflare.com/d1/).

## 5. R2 (Object Storage)

**What it is.** S3-compatible object storage with **zero egress fees**. Access via Workers binding, S3 API, or `r2.dev` public buckets. Standard storage + Infrequent Access tier. ([developers.cloudflare.com/r2](https://developers.cloudflare.com/r2/))

**S3 compatibility.** Works with the AWS SDK (`@aws-sdk/client-s3`) when you point it at the R2 endpoint and use R2 access keys. Also supports presigned URLs the same way. The Workers binding (`env.BUCKET.get/put/list/delete`) avoids the SDK and credential plumbing entirely.

**Lifecycle rules.** Yes — `PutBucketLifecycleConfiguration` (Class A op). Auto-transition to Infrequent Access or auto-delete after N days.

**Pricing (April 2026).**

| Tier | Storage | Class A (writes) | Class B (reads) | Egress |
|---|---|---|---|---|
| Standard | $0.015/GB-mo | $4.50/M | $0.36/M | **$0** |
| Infrequent Access | $0.01/GB-mo | $9.00/M | $0.90/M | $0.01/GB retrieval |
| Free tier | 10 GB-mo | 1M Class A | 10M Class B | $0 |

**Migration vs S3.** This is **the** big-ticket item. S3 charges $0.09/GB egress to the internet ($90/TB). For Gigler, if gigs produce large artifacts (videos, scraped datasets, build outputs) that users download, R2 saves potentially thousands per month. Use **Sippy** for incremental migration (pulls objects on-demand from S3 to R2 as they're requested).

**Migration gotchas.**

- Multipart upload threshold differs slightly (R2 doesn't enforce 5 MiB minimum part size for last part, etc.) — most SDKs work unchanged.
- `DeleteObject` and `AbortMultipartUpload` are free.
- Class A ops are billed per call: bursty writers (e.g., a sandbox writing 10k tiny files) get expensive — batch with `PutObject` of larger files when possible.

**Sources:** [R2 pricing](https://developers.cloudflare.com/r2/pricing/), [R2 docs](https://developers.cloudflare.com/r2/).

## 6. Cloudflare Email Service (Routing + Sending) ⚠ BETA

**Status.** Email Service is the umbrella product; **Email Routing** (inbound) is GA and free. **Email Sending** (outbound) is in **public beta** as of April 2026 — usable, but pricing is "TBC" and APIs may shift. Final pricing not announced. ([Email Service docs](https://developers.cloudflare.com/email-service/))

**What it does.**

- **Inbound (Routing)**: Catch-all + custom addresses on your domain. Route to other mailboxes, dispatch to a Worker, or to an Agent via `routeAgentEmail()`. Free.
- **Outbound (Sending)**: REST API (`POST /accounts/{id}/email/sending/send`) + Workers binding (`env.EMAIL.send({ to, from, subject, html, text })`). Supports CC/BCC, attachments, custom headers, delivery status.

**DNS / SPF / DKIM / DMARC.** Cloudflare must be your authoritative nameserver. Email Routing setup auto-provisions MX records and the necessary SPF/DKIM records — including DKIM key rotation. DMARC you set yourself (Cloudflare publishes guidance). Domains added to Email Sending get auto-configured DKIM signing for outbound.

**Worker integration.**

```ts
export default {
  async fetch(req, env) { /* ...send email via env.EMAIL.send(...) */ },
  async email(message, env) {
    // Inbound — dispatch to an agent
    await routeAgentEmail(message, env, { resolver: createAddressBasedEmailResolver("EmailAgent") });
  },
} satisfies ExportedHandler<{ EMAIL: SendEmail }>;
```

`wrangler.jsonc`:
```jsonc
{
  "send_email": [{ "name": "EMAIL" }],
  "email": [{ "name": "EMAIL_HANDLER" }]
}
```

**Agent integration.** See Section 2 — `onEmail`, `replyToEmail`, `sendEmail`, `forward`, `setReject` are all on the Agent base class. **Secure reply routing** via `createSecureReplyEmailResolver(secret)` HMAC-signs `X-Agent-Name`/`X-Agent-ID` headers and verifies inbound replies, preventing attackers from forging the reply target. `replyToEmail()` enforces the secret on routes that came in via the secure resolver.

**Migration gotchas vs SES.**

- **Reputation/IP warmup.** Cloudflare manages shared IPs; no dedicated IP option as of April 2026.
- **Pricing risk.** Email Sending pricing is TBC. SES is $0.10 per 1,000 emails — budget for similar pricing but treat as unknown.
- **Inbound is dramatically simpler than SES + Lambda + S3 storage of MIME.** This alone is worth migrating for.
- **Region/data residency** — Email Service runs on Cloudflare's global network. Check compliance posture.

**Sources:** [Email Service](https://developers.cloudflare.com/email-service/), [Email Routing](https://developers.cloudflare.com/email-routing/), [Agents Email API](https://developers.cloudflare.com/agents/api-reference/email/).

## 7. Browser Run (formerly Browser Rendering)

**What it is.** Headless Chrome on Cloudflare's global network. Use via:

- **Quick Actions** (no code): screenshots, PDFs, scrape, JSON extraction, crawl, links, snapshots — single HTTP requests.
- **Browser Sessions**: full control via Puppeteer (Cloudflare fork), Playwright (Cloudflare fork), CDP WebSocket, or Stagehand (AI-driven, intent-based selectors).
- **Playwright MCP**: lets LLMs control browsers via Model Context Protocol.

([developers.cloudflare.com/browser-rendering](https://developers.cloudflare.com/browser-rendering/))

**CDP endpoint.** Yes — connect from any environment (CI/CD, local) over WebSocket using your Cloudflare token. `puppeteer.connect({ browserWSEndpoint })` works.

**Live View / Human-in-the-Loop.** Live View streams the browser frame to a UI so a human can watch (and intervene). Useful for debugging agent runs and HITL approvals during sensitive automation.

**Session reuse / persistence.** Persist sessions across requests. Cloudflare specifically recommends pairing Browser Run with Durable Objects to keep a browser session pinned to a DO for the life of a gig.

**Limits.**

- **Free**: 3 concurrent browsers, 10 minutes/day.
- **Paid**: **120 concurrent browsers**, 10 hours/month included, 1 new browser/sec, 60s default timeout.
- Higher concurrency on case-by-case basis.

**Pricing (Paid).**

- 10 browser-hours/month included; **$0.09/additional browser-hour**.
- 10 concurrent-browsers included (monthly avg of daily peak); **$2.00/additional concurrent browser**.

**Migration gotchas.**

- vs AWS Lambda + headless Chromium layer: Cloudflare manages the Chromium binary, breakage on Chrome upgrades, and global pop placement.
- vs Browserbase / Browserless: Cloudflare is competitive on price, native bindings to R2/D1/Workers AI, but lacks some session-recording/replay features Browserbase has shipped.
- Iframe content is not accessible in Quick Actions / cursor-ide-browser MCP — same limitation applies to Browser Run via standard CDP rules around cross-origin iframes (use full Playwright + frame() APIs).

**Sources:** [Browser Run docs](https://developers.cloudflare.com/browser-rendering/), [Browser Run pricing](https://developers.cloudflare.com/browser-rendering/pricing/), [Browser Run limits](https://developers.cloudflare.com/browser-run/limits/).

## 8. Sandboxes (GA April 13, 2026)

**Status.** **Generally available as of April 13, 2026** ([Sandboxes GA blog](https://blog.cloudflare.com/sandbox-ga/)). Containers also went GA on the same date.

**What it is.** A persistent, isolated container per sandbox name, accessed via the `@cloudflare/sandbox` package. `getSandbox(env.Sandbox, "agent-session-47")` either gets the running sandbox or starts a fresh one. Idle sandboxes sleep automatically and wake on request. Powered by Cloudflare Containers + Durable Objects.

**Capabilities (all production-ready):**

- **`exec(cmd, args, { stream })`** — run shell commands with streaming output.
- **PTY support** (Feb 2026) — full pseudo-terminal over WebSocket, xterm.js compatible. `sandbox.terminal(request, { cols, rows })`.
- **Persistent code interpreters** — Python/JS/TS contexts where variables persist across `runCode()` calls (Jupyter-style). Returns charts, tables, structured outputs.
- **Background processes** — start servers, get readiness checks, get `preview URLs` to share with users mid-conversation.
- **File operations** — `mkdir`, `writeFile`, `readFile`, `gitClone`/`gitCheckout`.
- **File watching** — native filesystem events for hot reload.
- **Snapshots** — quick save/restore of sandbox state.
- **WebSocket connections** — `sandbox.wsConnect(req, port)` to reach servers in the sandbox.
- **Bucket mounting** — mount R2/S3/GCS as a local filesystem.
- **Secure credential injection** — programmable egress proxy. The agent never sees credentials; the proxy injects them per request based on host. This is the big security primitive that beats raw VMs.

```ts
class GigCoder extends Sandbox {
  static outboundByHost = {
    "api.github.com": (req, env) => {
      const headers = new Headers(req.headers);
      headers.set("authorization", `Bearer ${env.GITHUB_TOKEN}`);
      return fetch(req, { headers });
    },
  };
}
```

**Pricing.** **Active-CPU pricing**: **$0.000020 / vCPU-second** (= ~$72/month at 100% CPU on 1 vCPU). Free tier: 375 vCPU-minutes/month. You also pay underlying Workers + Durable Objects + Workers Logs. The big win vs Lambda: no charge for idle wall-clock time on a long-running sandbox waiting on user input.

**Migration vs AWS Lambda + Fargate / EC2.**

- Lambda: 15-min hard cap, no PTY, no persistent state — wrong abstraction for "agent that codes for hours."
- Fargate/EC2: pay for idle, no built-in egress proxy, no per-request billing.
- Sandboxes hit the sweet spot for agentic coding.

**Limits.** See [Sandbox limits](https://developers.cloudflare.com/sandbox/platform/limits/) — concurrency and CPU caps depend on plan; designed to scale "a fleet of agents."

**Gotcha.** Production deployment is required to use bucket mounting (not in `wrangler dev`).

**Sources:** [Sandbox docs](https://developers.cloudflare.com/sandbox/), [Sandbox SDK GitHub](https://github.com/cloudflare/sandbox-sdk), [Sandboxes GA blog](https://blog.cloudflare.com/sandbox-ga/), [Sandbox auth blog](https://blog.cloudflare.com/sandbox-auth).

## 9. Auth — Migrating off AWS Cognito

There is no direct Cognito replacement in Cloudflare's first-party stack. Three viable patterns:

### a) Cloudflare Access (zero-trust, app-level)
**Best for:** Internal tools, admin dashboards, B2B apps where users have IdP accounts (Google, Okta, GitHub, etc.). Access sits in front of your app as an identity-aware proxy and verifies JWT on each request. Not suitable for B2C signup flows. ([Access apps](https://developers.cloudflare.com/cloudflare-one/applications/))

### b) Better Auth on Workers (recommended for B2C)
Better Auth is the most idiomatic option in April 2026 — runs on Workers with D1 (Drizzle adapter) + KV for sessions. Critical caveat from production usage: **avoid dual auth instances** (singleton + per-request). Doing both causes 33-second hangs, 503s, and dropped sessions. Use a single per-request instance and `waitUntil` for background tasks. ([Better Auth + Workers integration guide](https://medium.com/@senioro.valentino/better-auth-cloudflare-workers-the-integration-guide-nobody-wrote-8480331d805f))

### c) Clerk (managed, easiest migration)
Clerk has a first-class **Cognito password migrator**: set `password_hasher: "awscognito"` and `password_digest: "awscognito###..."` on imported users. Requires enabling `ALLOW_USER_PASSWORD_AUTH` on a public Cognito client. Clerk verifies the password on first login by calling Cognito, then re-hashes with bcrypt. ([Clerk Cognito migration](https://clerk.com/docs/guides/development/migrating/cognito))

### Cognito password migration constraints
**Cognito does NOT export password hashes.** This is the universal Cognito migration pain point. Two options:
1. **Trampoline pattern** (Clerk does this) — keep Cognito running, verify on first login, re-hash on your platform.
2. **Force password reset** — email all users a reset link on cutover. Simpler but worse UX.

### Lucia
Worth noting that Lucia (the auth library) was **deprecated in March 2025**. Don't pick it for new work. Better Auth is the spiritual successor.

**Recommendation for Gigler.** If you have <10k users and want a managed solution, Clerk + the Cognito migrator is the lowest risk. If you want to own the stack on Cloudflare primitives, Better Auth + D1 + KV is the modern choice.

## 10. Workers AI vs Google Gemini

**Can you keep using Gemini directly via `fetch` from a Worker?** **Yes**, fully. Workers run plain `fetch()` to `generativelanguage.googleapis.com`. The Agents SDK explicitly supports OpenAI/Anthropic/Gemini/Workers AI as model providers ([Using AI models](https://developers.cloudflare.com/agents/api-reference/using-ai-models/)). You can even route Gemini calls through **AI Gateway** for caching, rate limiting, retries, fallback, and unified observability.

**Workers AI competitive models (April 2026):**

| Capability | Workers AI option | Gemini equivalent |
|---|---|---|
| General/agent | **GLM-4.7-Flash** (131k ctx), **Kimi K2.5** (256k ctx, multi-turn tools) | Gemini 1.5 Flash / Pro |
| Long context | Gemma 4 26B A4B (256k ctx), Kimi K2.5 (256k) | Gemini 1.5 Pro (2M) |
| Vision | Llama 4 Scout 17B, Gemma 4 26B A4B (OCR, charts, UI), Kimi K2.5 | Gemini 1.5 multimodal |
| Function calling | Native, via `@cloudflare/ai-utils` (embedded function calling) or standard tool-use | Gemini structured output |

**Gemini still wins for:** 2M-token context, native video/audio multimodal, raw price/perf at the very top end.

**Workers AI wins for:** Zero-egress to your Worker, no separate API key/billing relationship, AI Gateway out of the box, models co-located with R2/D1.

**Recommendation.** Keep Gemini for the heavy reasoning/long-context tasks via `fetch`+ AI Gateway. Use Workers AI for cheap classification, embeddings (`@cf/baai/bge-large-en-v1.5`), small tool-use loops, and anything where 200ms latency to a co-located model matters.

**Sources:** [Workers AI](https://developers.cloudflare.com/workers-ai/), [Models catalog](https://developers.cloudflare.com/workers-ai/models/), [AI Gateway](https://developers.cloudflare.com/ai-gateway/), [Function calling](https://developers.cloudflare.com/workers-ai/features/function-calling).

## 11. Queues + Workflows (vs SQS + Step Functions)

### Queues
Per-Worker producer/consumer messaging with batching, retries, delays, dead-letter queues, and HTTP pull consumers. ([Queues docs](https://developers.cloudflare.com/queues/))

**Pricing.** **$0.40/million operations**. An operation = 64 KB written/read/deleted. Most messages = 3 ops (write + read + delete). 1M operations included on Paid; ~10k/day free.

**vs SQS.** SQS = $0.40/million requests for Standard, but charges egress on payload retrieval. Queues includes egress ($0). Queues is also tighter integrated with Workers (no IAM, no SDK plumbing). DLQ + delays are built in. Pull consumers exist for non-Workers consumers via HTTP.

### Workflows
Durable, multi-step execution with `step.do()`, `step.sleep()`, `step.sleepUntil()`, `step.waitForEvent()`, automatic retries, and persistence across weeks. ([Workflows docs](https://developers.cloudflare.com/workflows/))

**Pricing.** Same as Workers (10M req + 30M CPU-ms included on Paid). Storage 1 GB included, then $0.20/GB-month.

**Limits (Paid).**

- 10,000 steps default per workflow (configurable to 25k).
- 30s active CPU per step (configurable to 5 min).
- **Wall-clock duration unlimited** — `step.sleep()` can wait up to **365 days** without burning compute.
- Free plan: 1024 steps, 10ms CPU per step.

**vs Step Functions.**

| Feature | Step Functions | Cloudflare Workflows |
|---|---|---|
| Definition | JSON/YAML state machine (ASL) | TypeScript code (just `step.do(...)`) |
| Local dev | Limited (sam local, Step Functions Local) | First-class via wrangler |
| Pricing | $0.025/1k state transitions | Bundled into Workers pricing |
| Max wait | 1 year | 1 year |
| Human-in-loop | `waitForTaskToken` callback pattern | `step.waitForEvent()` + Agent `waitForApproval()` |
| Compute integration | Lambda, ECS, etc. (network calls) | Workers + DO + R2 + D1 (bindings) |

**Recommendation.** Workflows is materially better DX than Step Functions for code-first teams. Use Queues for fan-out and async work; Workflows for any operation that has > 1 step or needs to sleep.

## 12. Twilio Integration from Workers

**Confirmed: Twilio webhooks point at a Workers URL with no friction.**

- Twilio sends `application/x-www-form-urlencoded` POSTs (SMS) or `application/json` (Conversations). Workers handles both via standard `request.formData()` / `request.json()`.
- **Signature validation**: read `X-Twilio-Signature` header, compute HMAC-SHA1 of the URL + sorted form params using your auth token via Web Crypto API, compare with `crypto.subtle.timingSafeEqual` semantics.
- **Critical gotcha**: the verification function should not be `async` returning a boolean wrapped in a promise — an unawaited promise is always truthy and fails open. Wait the result.
- Cloudflare has a [GitHub-SMS-via-Twilio tutorial](https://developers.cloudflare.com/workers/tutorials/github-sms-notifications-using-twilio/) and the Agents SDK has a [generic webhooks guide](https://developers.cloudflare.com/agents/guides/webhooks/) that covers this exact pattern.

Twilio Conversations webhooks (group chat, multi-channel) work identically — just a different payload shape. There is no Workers-specific incompatibility.

## 13. Pricing Comparison (Cloudflare vs AWS)

Best-effort comparison for a moderately-sized Gigler deployment. All numbers are list, no committed-use discounts.

### Per-product

| Product | Cloudflare | AWS equivalent |
|---|---|---|
| **Compute (Worker)** | $5/mo + 10M req included; $0.30/M overage; 30M CPU-ms included; $0.02/M ms overage | Lambda: $0.20/M req + $0.0000167/GB-s; API Gateway: $1.00/M req |
| **Stateful actor (DO)** | $0.15/M req + $12.50/M GB-s; 1M+400k GB-s included | DynamoDB streams + Lambda: roughly equivalent compute, no built-in WebSocket/state |
| **SQL** | D1: $0.001/M reads, $1.00/M writes, $0.75/GB-mo | DynamoDB on-demand: $0.25/M reads, $1.25/M writes, $0.25/GB-mo |
| **Object storage** | R2: $0.015/GB-mo, **$0 egress** | S3: $0.023/GB-mo, **$0.09/GB egress** |
| **Email inbound** | Free | SES: $0.10/1k inbound + Lambda + S3 storage |
| **Email outbound** | Email Sending (beta, TBC pricing) | SES: $0.10/1k |
| **Browser** | $0.09/browser-hour, $2/extra concurrent | Lambda + headless Chromium layer (manual); Browserless ~$200/mo entry |
| **Sandbox/exec** | $0.000020/vCPU-second active CPU | Fargate: $0.04048/vCPU-hour wall-clock (~5x more for active workloads, much more for idle) |
| **Workflows** | Bundled in Workers pricing | Step Functions: $0.025/1k state transitions |
| **Queues** | $0.40/M operations | SQS: $0.40/M req + egress |
| **Auth** | Build on Better Auth + D1 (no per-MAU fee) | Cognito: $0.0055/MAU after 50k free |

### Where Cloudflare wins big for Gigler

1. **R2 zero egress** — every gig that produces a downloadable artifact saves real money.
2. **Sandbox active-CPU pricing** — long-running coding sessions where the sandbox is mostly waiting on the agent or user are nearly free vs Fargate.
3. **DO compute model** — per-tenant isolation without paying for an idle VM.
4. **Bundled Workflows** — vs Step Functions per-transition fees that add up fast in agentic workloads with many steps.
5. **No NAT Gateway, no VPC, no IAM permission boundaries** — operational simplicity.

### Where AWS still wins

1. **DynamoDB** at very high scale with global tables and DAX caching is hard to beat operationally.
2. **SES** has higher sender reputation and a known brand; CF Email Sending is beta.
3. **Compliance breadth** — SES/S3 have more attestations (HIPAA BAA, FedRAMP High, etc.) than R2/Email Service today.
4. **Cognito** offers SAML/OIDC IdP federation for enterprise B2B at managed-service price; Better Auth requires you to build/operate.

## Risk Register: "Don't bet your business on this yet"

| Item | Risk | Mitigation |
|---|---|---|
| **Cloudflare Email Sending** ⚠ BETA | Pricing TBC, deliverability vs SES unknown at scale | Keep SES as fallback for first 90 days; A/B sender domains |
| **Agents SDK fibers/sub-agents** | Fast-moving API surface (April 2026 fiber refactor; sub-agents not yet first-class) | Pin `agents` version, gate on changelog before upgrading; avoid sub-agent patterns for critical paths until Project Think GA |
| **D1 row-scan billing** | Unindexed queries can blow up costs silently | Enforce indexes via lint/CI; monitor `meta.rows_read` |
| **DO storage billing started Jan 7, 2026** | Net-new line item not in older estimates | Re-run cost models with current pricing |
| **OpenNext + Next.js Node middleware** | Not yet supported (Next 15.2+) | Audit middleware for Node-only APIs before migrating |
| **Browser Run concurrent caps** | 120 concurrent on standard Paid; bursts beyond require account-level escalation | Architect with DO-pinned reuse; pre-warm pool via `scheduleEvery` |
| **No Cognito password export** | First-time login must trampoline through Cognito | Use Clerk's migrator or build the trampoline yourself |

## Suggested Migration Sequencing for Gigler

1. **R2 first** (zero risk, immediate egress savings) — use Sippy to backfill from S3 lazily.
2. **D1 + DO SQLite** for new features in parallel with DynamoDB; start moving cold/secondary access patterns.
3. **Workers + OpenNext** for the Next.js app — deploy alongside Amplify on a preview subdomain, cut over via DNS.
4. **Email Routing** (free, GA) immediately for inbound; defer Email Sending until it leaves beta or accept the risk for transactional mail.
5. **Sandboxes** for the "real coding gig" feature — this is greenfield; nothing to migrate.
6. **Browser Run** for the "real browsing gig" feature — same.
7. **Agents SDK** as the orchestration backbone, with Workflows for HITL.
8. **Cognito → Clerk (managed) or Better Auth (own it)** — last, because auth migrations carry the highest user-facing risk.
