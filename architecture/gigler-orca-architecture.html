GIGLER
Overview

A completed gig is the product — not a chat transcript.

Users send Gigler an outcome they want over the channels they already use — SMS, email, or voice. Orca, Gigler's orchestration layer, decomposes the request into steps, routes each step to the right model, agent, tool, or platform, tracks state, pauses for human approval only when needed, and delivers a real artifact through a shareable link.

The input

Channels people already live in

No app to install, no dashboard to learn. Text it, email it, or call it. Each gig even gets its own email address.

The brain

Orca orchestration

Plan → route → execute → verify. Picks the cheapest capable tool first; escalates to real browsers, sandboxes, or a human only when required.

The output

Deliverables, not answers

A deployed site, a designed PDF, a submitted form, a booked reservation — delivered as a gigler.ai/abc123 link.

This document describes the current production system (AWS) and the email-native evolution on Cloudflare that hardens Gigler for production scale.

Email-native agents

Every gig is an agent with its own email address

Email is the most universal interface on earth. Using Cloudflare's Email Service + Agents SDK, each gig gets a unique sub-address that routes inbound mail straight to that gig's agent instance. The inbox becomes the agent's memory.

Inbound email → gig+1a2b3c@gigler.ai
Cloudflare Email Routing receives all *@gigler.ai
▼
routeAgentEmail() + address resolver
sub-address maps to the exact gig agent instance
▼
GigAgent.onEmail()
parse, persist to state, kick off Orca work, reply / escalate async
▼
env.EMAIL.send() — async reply
HMAC-signed reply routing back to the same agent
export class GigAgent extends Agent { async onEmail(email: AgentEmail) { const msg = await PostalMime.parse(await email.getRaw()); // inbox = memory; persists in Durable Object this.setState({ ...this.state, lastFrom: email.from }); // hand the request to Orca, run async await this.runFiber("orca", () => orca.handle(msg)); } } export default { async email(message, env) { await routeAgentEmail(message, env, { resolver: createAddressBasedEmailResolver("GigAgent"), }); }, };
  • No inbox provisioning — the address itself is the route
  • Asynchronous work — receive now, process for an hour, reply when done
  • Secure replies — HMAC-SHA256 headers prevent reply hijacking
  • Same agent, every channel — SMS, voice & email share one gig state

State architecture

Per-gig agent state + cross-tenant index

One agent (Durable Object) per gig

Each gig is a physically isolated stateful actor with its own embedded SQLite, alarms, and live WebSocket to the dashboard.

  • messages — full conversation across all channels
  • participants — members & roles (group gigs)
  • reminders — durable alarms for follow-ups
  • media — pointers to R2 objects
  • checkpoints — durable-execution recovery state

Cross-tenant index (D1 / relational)

A small relational store for everything that must be queried across gigs and users.

  • User — phone + email identity, plan, billing
  • Gig — shortCode, owner, status, title
  • Deliverable — shortCode → R2 key (link lookup)
  • UserIntegration — encrypted third-party tokens
ConcernToday (AWS)Hardened (Cloudflare)
Per-gig live stateDynamoDB shared tablesDurable Object SQLite — physical isolation per gig
Cross-gig queries / billingDynamoDB + GSIsD1 (managed SQLite), indexed
Media & deliverablesS3 + CloudFrontR2 — zero egress, S3-compatible
RemindersEventBridge scan every 5 minPer-gig durable alarms (no polling)
Dashboard updatesPollingLive WebSocket state sync

Execution surfaces

The tools Orca routes work to

Documents & media

HTML → Chrome → PDF / PNG

Agent writes designed HTML, headless Chrome prints a pitch deck, report, invoice, menu, or MMS card. Source HTML stored for one-line iteration ("make it blue").

Browser tasks

Real Chrome + HITL

Form-fill, bookings, bill pay, applications. Dismisses cookie walls, handles SPAs, captures proof screenshots. Hands off to a human for logins & CAPTCHAs.

Code & deploy

Sandbox containers

Persistent containers scaffold, run, and deploy a site. Live preview URL streamed to the dashboard. Credentials injected at the egress proxy — agent never sees them.

Vision & reasoning

Frontier LLM (model-agnostic)

Reads PDFs, receipts, and screenshots; drafts copy; classifies intent; calls functions. The reasoning core behind every gig — routed to the best-fit model via a gateway.

Third-party platforms

Adapter pattern

OpenTable, Resy, Evite, Stripe behind a uniform search / execute / confirm / cancel interface. Always confirms with the user before finalizing.

Scheduling & voice

Reminders + realtime voice

Durable reminders, wake-up calls, check-ins. Real-time speech-to-speech voice for hands-free gig updates.

Every action that touches the outside world is captured (screenshot + metadata) as irrefutable proof of work, stored and searchable from the dashboard.

Orchestration model

Model-agnostic orchestration, aligned with ADK & MCP

AI across the gig lifecycle

  • Intent & gig-type classification on every inbound message
  • Multi-step reasoning & planning inside the Orca loop
  • Vision — reads PDFs, receipts, screenshots, uploaded photos
  • Generation — drafts copy, designed HTML, and images
  • Function-calling — the tool interface Orca routes through
  • Realtime voice — speech-to-speech for hands-free gigs

Agent patterns & portability

  • MCP — tools (email, browser, code) exposed as Model Context Protocol servers Orca can securely call
  • ADK-aligned — declarative agent + tool definitions; Orca maps cleanly onto the Agent Development Kit pattern
  • Model gateway — a thin gateway routes each step to the best-fit frontier model; no lock-in to any single lab or provider
  • Google Cloud ready — credits applied to evaluation, optimization, and a path toward Marketplace listing

Orca is not "another chat model." It is an orchestration layer that selects and coordinates the right model, agent, tool, and platform per step — provider-agnostic by design — and improves its routing as more gigs complete.

GIGLER
Reliability & trust

Production hardening & safety

Durable execution

Nothing gets lost

Long gigs checkpoint and resume after eviction, timeouts, or a closed laptop — the agent picks up from the last completed step.

Human-in-the-loop

Control where it counts

Logins, payments, irreversible submits, and pre-fill confirmations pause for explicit approval with escalation reminders.

Email safety

No loops, no hijacks

RFC 3834 auto-reply guards prevent mail loops; HMAC-signed reply routing prevents header forgery to other agents.

Credential isolation

Secrets stay secret

Third-party tokens injected at the egress proxy; the model and sandbox code never see raw credentials.

Cost guardrails

No runaway agents

Per-gig browser-minute caps, idle-session shutdown, and plan-tier budgets enforced before any expensive tool launches.

Proof of work

Verifiable outcomes

Every external action stores a confirmation screenshot + metadata, searchable from the dashboard — trust by evidence.
